**Service:** Matching Engine
**Version:** V1.0

**Last Updated:** July 2026

# 1. Purpose

This document defines the exact Kafka event payloads the Matching Engine consumes and publishes, so that Order Service, Settlement Service, and any future consumer can integrate against a fixed contract rather than inferring shapes from code.

All ID fields follow `TradeDrift_ID_Correlation_Standard.md` — UUIDv7, generated by the owning service, propagated unchanged.

---

# 2. Event Flow Overview

## 2.1 High-Level Event Map

```
Order Service ─── OrderCreated ─────────────────────► Matching Engine
Order Service ─── OrderCancelRequested ─────────────► Matching Engine
                                                               │
                                     ┌─────────────────────────┴────────────────────────┐
                                     │                                                   │
                              TradeExecuted                                      OrderCancelled
                           (0 to many per                                     (0 or 1 per
                            OrderCreated)                                 OrderCancelRequested)
                                     │                                                   │
                                     ▼                                                   ▼
                           Settlement Service                                  Order Service
                         (calls SettleTrade)                          (transitions order state,
                                                                        releases reserved funds)
```

---

## 2.2 Event Publishing Flow — OrderCreated

```
OrderCreated received
        │
        ▼
Pre-match checks pass?
        │
        ├── No (invalid tick/lot size)
        │       │
        │       ▼
        │   OrderCancelled {reason: "invalid_order_parameters"}
        │
        └── Yes
                │
                ▼
          Match Loop runs against opposite side
                │
                ├── Fills produced ──────────────► TradeExecuted × N  (one per fill)
                │
                ├── LIMIT, remaining qty > 0 ───► Insert as resting order  (no event)
                │
                └── MARKET, remaining qty > 0 ──► Remainder discarded IOC  (no event)
```

---

## 2.3 Event Publishing Flow — OrderCancelRequested

```
OrderCancelRequested received
        │
        ▼
orderIndex lookup
        │
        ├── Order found and removed from book
        │       │
        │       ▼
        │   OrderCancelled
        │   { order_id, user_id, market_id,
        │     remaining_quantity, reason: "user_requested",
        │     cancelled_at }
        │
        └── Order not found
                │
                ▼
            No event published
            (already fully filled or ID unknown — silent idempotent no-op)
```

# 3. Events Consumed

## 3.1 `OrderCreated`

Published by Order Service via its Outbox (`07_Order_Service.md`). Partition key: `market_id`.

| Field | Type | Notes |
| --- | --- | --- |
| `order_id` | UUID (v7) | Generated by Order Service before persistence |
| `user_id` | UUID (v7) | Order owner |
| `market_id` | string | e.g. `BTC-USDT`; Kafka partition key |
| `side` | string | `BUY` \| `SELL` |
| `order_type` | string | `LIMIT` \| `MARKET` |
| `price` | decimal string | Present for `LIMIT`; **field omitted for `MARKET`** (not null — field absent entirely) |
| `quantity` | decimal string | Original order quantity |
| `created_at` | RFC3339 timestamp | Order Service acceptance time — informational only; ME sets its own `timestamp` on the `OrderNode` at insertion (`04_Data_Structures/03_Order_Node.md`) |

The ME maps this payload directly onto an `OrderNode`: `orderID`, `userID`, `marketID`, `side`, `price`, `originalQty` (= `quantity`), `remainingQty` (initialized = `quantity`). See `04_Data_Structures/03_Order_Node.md §3`.

---

## 3.2 `OrderCancelRequested`

Published by Order Service on the **same partition key** (`market_id`) as `OrderCreated`, guaranteeing per-market ordering between creates and cancels (`07_Order_Service.md §Cancel vs Fill Race Condition`).

| Field | Type | Notes |
| --- | --- | --- |
| `order_id` | UUID (v7) | The resting order to cancel |
| `market_id` | string | Redundant with partition key; included for consumer convenience and logging |
| `user_id` | UUID (v7) | For audit logging; ME does not re-authorize — Order Service already checked ownership |
| `requested_at` | RFC3339 timestamp | Informational |

---

# 4. Events Published

## 4.1 `TradeExecuted`

Published once per fill — a single incoming order can produce multiple `TradeExecuted` events in one sweep (`05_Matching_Algorithm.md §5`). Consumed by Settlement Service, which calls `Wallet.SettleTrade(...)`.

| Field | Type | Notes |
| --- | --- | --- |
| `trade_id` | UUID (v7) | **Generated by the Matching Engine**, in-memory, at match time (`02_System_Architecture.md §12`). Idempotency key for `SettleTrade`. |
| `market_id` | string | Denormalized from the `OrderNode` |
| `buy_order_id` | UUID (v7) | Whichever of maker/taker had `side == BUY` (`05_Matching_Algorithm.md §8`) |
| `sell_order_id` | UUID (v7) | Whichever of maker/taker had `side == SELL` (`05_Matching_Algorithm.md §8`) |
| `buyer_id` | UUID (v7) | `user_id` of the buy-side order |
| `seller_id` | UUID (v7) | `user_id` of the sell-side order |
| `maker_order_id` | UUID (v7) | The resting order consumed |
| `taker_order_id` | UUID (v7) | The incoming order that caused the match |
| `base_asset` | string | e.g. `BTC` — derived from `market_id` at startup; not split on every event |
| `quote_asset` | string | e.g. `USDT` — derived from `market_id` at startup |
| `price` | decimal string | Always the **maker's** price (`05_Matching_Algorithm.md §9`) |
| `quantity` | decimal string | Fill quantity for this trade: `min(incoming.remainingQty, best.remainingQty)` |
| `executed_at` | RFC3339 timestamp | ME wall-clock time when the fill was produced |

> **Field-for-field match with `SettleTrade`:** Wallet Service's signature is `SettleTrade(trade_id, buyer_id, seller_id, buy_order_id, sell_order_id, base_asset, quote_asset, price, quantity)`. Every one of those 9 parameters is present verbatim in this payload — Settlement Service performs no field translation, only pass-through plus its own outbox write.

**Both `buy_order_id`/`sell_order_id` and `maker_order_id`/`taker_order_id` are included.** They serve different consumers: Settlement/Wallet need buy/sell semantics (which leg gets which asset); Order Service bookkeeping and any future maker/taker fee logic need maker/taker semantics. Carrying both avoids forcing a consumer to re-derive one from the other.

---

## 4.2 `OrderCancelled`

Published when a cancel is applied to the book, or when an order is rejected by a pre-match check. §5 covers when it is *not* published.

Consumed by Order Service, which transitions the order to `CANCELLED` and calls `Wallet.ReleaseFunds(order_id)`.

| Field | Type | Notes |
| --- | --- | --- |
| `order_id` | UUID (v7) | The cancelled order |
| `user_id` | UUID (v7) | The order owner — used by Order Service to route and by Wallet Service to release the correct user's reserved funds |
| `market_id` | string | Denormalized |
| `remaining_quantity` | decimal string | Quantity resting at time of cancellation — Wallet Service releases exactly this amount (`07_Order_Service.md §Failure Handling`) |
| `reason` | string | `"user_requested"` or `"invalid_order_parameters"` — lets Order Service distinguish a user cancel from a system rejection for UX and audit |
| `cancelled_at` | RFC3339 timestamp | ME wall-clock time cancel was applied |

# 5. Events the Matching Engine Intentionally Does NOT Publish

| Event | Published by ME? | Actual source |
| --- | --- | --- |
| `OrderAccepted` | No | Not published by any service — acceptance is implicit in `OrderCreated` being committed |
| `OrderPartiallyFilled` | **No** | Order Service derives this from a `TradeExecuted` referencing its `order_id` |
| `OrderFilled` | **No** | Order Service derives this from a `TradeExecuted` that fully consumes the order |

The Matching Engine publishes **exactly two event types**: `TradeExecuted` and `OrderCancelled`. Order Service's own state machine is responsible for turning those signals into its richer `OPEN → PARTIALLY_FILLED → FILLED` lifecycle.

When a `MARKET` order finds no liquidity (`05_Matching_Algorithm.md §6`), **no event is published** — this is not an error, just an empty IOC result.

---

# 6. Cancel-vs-Fill Interaction

Fills always take precedence over cancels (`07_Order_Service.md §Cancel vs Fill Race Condition`). The ME's contract reflects this directly:

| Book state when `OrderCancelRequested` processed | ME publishes |
| --- | --- |
| Order fully filled (already removed from book) | Nothing — `Cancel()` is a silent no-op |
| Order partially filled, remainder resting | `OrderCancelled` with `remaining_quantity` = qty left at time of cancel |
| Order never matched, fully resting | `OrderCancelled` with `remaining_quantity` = original `quantity` |

`Cancel()` is naturally idempotent — the `orderIndex` lookup already encodes "is this still cancellable." No explicit state check needed before calling it.

# 7. Envelope / Versioning

| Property | V1 Behaviour |
| --- | --- |
| Serialization | JSON — consistent with Order Service and Wallet Service; no binary schema registry in V1 |
| Schema evolution | New optional fields may be added without a version bump; consumers must ignore unknown fields |
| Breaking change | Removing or renaming a field requires a new topic version (e.g. `TradeExecuted.v2`) — not in scope for V1 |
| Ordering guarantee | All events for a given `market_id` are in-order — `OrderCreated`/`OrderCancelRequested` share partition key; `TradeExecuted`/`OrderCancelled` published by the same single-threaded Publisher Layer per market |
| Idempotency key | `trade_id` for `SettleTrade`; `order_id` + event type for Order Service status transitions |
