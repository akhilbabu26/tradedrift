# TradeDrift — Identifier & Correlation Standard

> **Status:** ✅ Designed (V2)
> **Scope:** A cross-service standard, referenced by every service design doc rather than restated in each one. Applies to Authentication, Order, Wallet, Settlement, Matching Engine, Trade, Portfolio, Market, and Notification services.
> **Revision notes:** V2 adds the Notification aggregate (`notification_id`) owned by the Notification Service.

## Purpose

Every aggregate in TradeDrift (User, Wallet, Order, Trade, Reservation, Transaction) needs a globally unique identifier that exists **before** the owning row is persisted, so that gRPC calls, Kafka events, logs, and traces referencing it are unambiguous from the very first step of its lifecycle — not just after a database assigns one.

## 1. Identifier Generation: UUIDv7

All entity IDs are generated as **UUIDv7**, not UUIDv4 or database auto-increment.

- UUIDv7 embeds a millisecond timestamp in its most significant bits, so IDs are naturally time-ordered — sorting by ID approximates sorting by creation time, useful for debugging, log correlation, and range queries.
- Unlike UUIDv4 (fully random), UUIDv7 improves index locality in PostgreSQL — sequential-ish inserts, less B-tree page fragmentation than pure-random UUIDs.
- Still globally unique and collision-safe like v4 — no coordination needed between services generating IDs independently.

## 2. Database Column Type: PostgreSQL UUID

All ID columns use the native `UUID` type — never `VARCHAR(36)` or `TEXT`.

```sql
id UUID PRIMARY KEY
```

- Smaller on-disk representation (16 bytes vs 36-character string).
- Validates UUID format at the database level for free — a malformed ID is rejected on insert, not discovered later.
- Indexes more efficiently than a string-typed equivalent column.

## 3. ID Ownership: Generated by the Owning Service, Before Persistence

Every ID is generated in application code by the service that owns that aggregate, before the first `INSERT` — never assigned by the database (no `BIGSERIAL`, `IDENTITY`, or `AUTO_INCREMENT` anywhere in the schema).

```
Generate UUIDv7 (order_id)
  ↓
ReserveFunds(order_id)   -- Wallet Service needs the ID before Order Service's
                          -- row exists
  ↓
Wallet Service confirms reservation
  ↓
Save Order(id = order_id) + Outbox row   -- one transaction
  ↓
Commit
```

> **Why this matters:** If `order_id` were only assigned after `INSERT` (e.g. `BIGSERIAL`), Order Service could not tell Wallet Service which order a reservation belongs to until after the row exists — but the reservation must exist before the order is considered accepted. Generating the ID first removes this ordering deadlock entirely.

## 4. Propagation: One ID, Reused Everywhere

The same ID value is reused, unchanged, across every layer that touches that aggregate — never translated, re-encoded, or replaced with a local synonym:

- gRPC requests and responses (field name matches across services, e.g. `order_id` everywhere, not `orderId` in one service and `order_ref` in another).
- Kafka event payloads and message keys / partition keys where applicable.
- Structured log fields (every log line touching an order includes `order_id`).
- Distributed tracing spans, once tracing is introduced (see [Section 6](#6-open-item-infra-level-trace-id-not-required-for-v1)).

## 5. Correlation: `order_id` as the Order Lifecycle Correlation Key

`order_id` is the correlation ID for the entire order lifecycle, from creation through settlement:

```
OrderCreated ──┐
OrderAccepted  │
OrderPartiallyFilled │  all carry order_id,
OrderFilled    │  so the full history of
TradeExecuted  │  a single order can be
TradeSettled   │  reconstructed by querying
OrderCancelled ┘  every topic for one ID
```

This matches Order Service's own saga design ("`order_id` is carried as a correlation ID on every event in the chain") — this document makes it a standard for **all** services, not an Order-Service-specific detail.

## 6. Open Item: Infra-Level Trace ID (Not Required for V1)

> **Decision (deferred):** `order_id` works as a business correlation key once an order exists, but some request paths precede any `order_id` — e.g. auth, or a gateway request that fans out to multiple orders. A separate, generic `trace_id` (e.g. via OpenTelemetry) would cover those gaps. Not required for V1; noted here as a conscious deferral, not an oversight, so it isn't rediscovered as a surprise later.

## 7. Standard Aggregate IDs

| Aggregate | ID field | Owning service |
|---|---|---|
| User | `user_id` | Authentication Service |
| Wallet | `wallet_id` | Wallet Service |
| Order | `order_id` | Order Service |
| Reservation | `reservation_id` | Wallet Service |
| Trade | `trade_id` | Matching Engine |
| Wallet Transaction | `transaction_id` | Wallet Service |
| Notification | `notification_id` | Notification Service |

Every service design doc should reference this document for ID generation rules rather than restating them.
