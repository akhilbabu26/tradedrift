## 1. REST API Design Standards

All public HTTP endpoints exposed by the API Gateway must adhere to these standard policies to maintain predictability and consistency.

### 1.1 Versioning Policy
* **Public URL Prefix:** All endpoints must be versioned. The initial release uses the `/api/v1` namespace (e.g. `/api/v1/orders`).
* **Breaking Changes:** Breaking modifications (such as dropping fields or changing datatypes) require incrementing the major version to `/api/v2`.
* **Backward Compatibility:** Non-breaking modifications (such as appending optional properties to responses or request bodies) do not trigger version changes. Existing properties are never repurposed.

### 1.2 Boundary Naming & Casing Translation
To enforce clear data conventions, names translate automatically at boundary layers:

```

      camelCase                           CamelCase                           snake_case
  e.g., "availableBalance"           e.g., "AvailableBalance"            e.g., "available_balance"
```

1. **REST JSON Payload:** Property names must use **`camelCase`** (e.g. `orderId`, `price`).
2. **PostgreSQL DB Tables:** Column names must use **`snake_case`** (e.g. `order_id`, `price`).
3. **Go Application Code:** Struct properties must use **`CamelCase`** (e.g. `OrderID`, `Price`).

### 1.3 Intentional Translation Mappings
Naming differences across boundaries (for example, client-facing `orderType` in REST API vs `order_type` inside Postgres columns) are intentional and expected mappings. These are handled natively by Go structs using field tags:
```go
type Order struct {
    OrderType string `json:"orderType" db:"order_type"`
}
```

### 1.4 Supported Assets Initialization
To guarantee that newly provisioned databases are instantly ready for registration:
* **Seeding Choice:** Standard supported assets (`USDT`, `BTC`, `ETH`, `SOL`) are initialized via SQL `INSERT` DDL statements at database setup time (SQL seed files / migrations) rather than application bootstrap code.
* **Dynamic Expansion:** Adding new tradable tokens post-deployment is managed by DBA transactions inserting new rows into the `supported_assets` table without requiring application code changes.

### 1.5 Event Schema Version Consistency
* All event formats published by the Outbox Pattern must follow strict version consistency matching:
  - Protobuf Package Version: `v1` (e.g. `order/v1/order.proto`).
  - Kafka Suffix Suffix: `.v1` (e.g. `orders.created.v1`).
  - Envelope Metadata: `event_version = 1`.
* Breaking updates require releasing a parallel `v2` package stub and a corresponding `.v2` Kafka topic pipeline.

## 2. Request Identification & Tracing Context

All incoming HTTP requests must map headers for tracing:
* **`X-Request-ID`:** A UUIDv7 generated by the client or the API Gateway. The value must be echoed back in the response headers.
* **`traceparent`:** Standard W3C distributed tracing context.
* **Echo Requirement:** If `X-Request-ID` is passed in the request header, the API Gateway must return it in the response header:
  ```http
  HTTP/1.1 200 OK
  Content-Type: application/json
  X-Request-ID: 018f673a-4e2b-7f11-80a2-c3bfde34aa5a
  ```

## 3. Idempotency Protocol

To prevent duplicate mutations during connection retries, state-modifying POST requests must require the **`Idempotency-Key`** header:
* **Target Endpoints:** `POST /api/v1/orders`, `POST /api/v1/wallets/deposits`, and `POST /api/v1/wallets/withdrawals`.
* **Key Format:** Standard UUID string (UUIDv4 or UUIDv7).
* **Storage & Expiration:** The API Gateway stores keys in Redis with a **24-hour expiration window** (`TTL 86400`).
* **Duplicate Execution Check:**
  - **Identical Payload:** If the key exists and the request body matches the cached transaction, the Gateway immediately returns the cached response without calling backend services.
  - **Conflict Payload Mismatch:** If the key exists but the request body parameters differ, the Gateway aborts and returns `409 Conflict` containing the `API_IDEMPOTENCY_CONFLICT` error code.

## 4. Query Pagination & Filtering Standards

### 4.1 Keyset Cursor Pagination
All list endpoints returning large arrays (such as order histories, trade logs, and notifications) must use keyset cursor pagination to avoid performance penalties.
* **Request Schema:**
  - `cursor`: A base64-encoded string containing pagination offsets (e.g., `eyJjcmVhdGVkX2F0IjoiMjAyNi0wNy0xMVQxMjo1MDowMFoiLCJpZCI6IjAxOGY2NyJ9`). If empty, starts from the newest records.
  - `limit`: The maximum records to fetch. Defaults to `20`, maximum `100`.
* **Response Schema:**
  ```json
  {
    "data": [],
    "nextCursor": "eyJjcmVhdGVkX2F0IjoiMjAyNi0wNy0xMVQxMjo0ODowMFoiLCJpZCI6IjAxOGY2OCJ9"
  }
  ```

### 4.2 Query Filtering Parameters
When filtering query lists, services must expose a subset of these standard parameters:
* `status`: Filters by state (e.g. `status=OPEN` or `status=FILLED`).
* `marketId`: Filters by pair (e.g. `marketId=BTC-USDT`).
* `side`: Filters by direction (e.g. `side=BUY`).
* `from` / `to`: ISO-8601 timestamps defining time bounds.

## 5. Gateway Rate Limits

The API Gateway enforces rate limits per client identity (User ID for authenticated sessions, IP address for guest endpoints):

| API Endpoint Scope | Limit Threshold | Action on Exceeded |
|---|---|---|
| **`/api/v1/auth/login`** | 5 requests / minute | `429 Too Many Requests` |
| **`/api/v1/orders`** (Placements / Cancels) | 20 requests / second | `429 Too Many Requests` |
| **`/api/v1/markets`** (Public metadata / Books) | 100 requests / second | `429 Too Many Requests` |
| **WebSocket Connection** | Max 50 active channel subscriptions | Reject subscription request |

## 6. Error Code Registry

All error responses must return standard JSON wrappers using cataloged string codes:

```json
{
  "errorCode": "INSUFFICIENT_FUNDS",
  "message": "User available balance is insufficient",
  "timestamp": "2026-07-11T12:00:00Z"
}
```

### Core Error Catalog

| Error Code | HTTP Status | Description |
|---|---|---|
| `AUTH_INVALID_CREDENTIALS` | `401 Unauthorized` | Login credentials mismatch. |
| `AUTH_ACCOUNT_LOCKED` | `403 Forbidden` | User locked out after 5 failed login attempts. |
| `AUTH_INVALID_TOKEN` | `401 Unauthorized` | Access token signature expired or revoked. |
| `WALLET_FROZEN` | `403 Forbidden` | Wallet is locked for the target asset. |
| `INSUFFICIENT_FUNDS` | `400 Bad Request` | Reservation request amount exceeds available balance. |
| `ORDER_NOT_FOUND` | `404 Not Found` | Target order ID does not exist. |
| `ORDER_ALREADY_FILLED` | `400 Bad Request` | Cancellation request rejected because order is already filled. |
| `MARKET_HALTED` | `503 Service Unavailable` | Order rejected because trading has been administratively halted. |
| `API_IDEMPOTENCY_CONFLICT` | `409 Conflict` | Same idempotency key used with different request parameters. |
| `API_RATE_LIMIT_EXCEEDED` | `429 Too Many Requests` | Request rate threshold exceeded. |
